## 1/7
## awsクラウディングの六つのメリット
-　固定費が柔軟な変動費へ : 
使った分だけ払う従量課金制なので設置などの固定費が要らなくなる。
- コストの最適化:
コストセンターへの運用と保守の投資が要らなくなり別のところのお金が回せる。
- キャパシティ予測が不要に:
オンプレミス環境だと予測に数ヶ月かかったがawsによって必要に応じてになった。状況に応じてスケーリングを行う。
- スケールによるコストメリット:
多くのユーザーによって規模の経済の利点を活かして使用量が増えることによってサービスごとの料金が下がることもある。
- スピードと俊敏制の向上:
オンプレミス環境ではリソースの導入に数ヶ月かかっていたものの、クラウドを使用することによって迅速なリソース調達ができる。アジャイル開発を利用して開発コストや時間が大幅に削減されて組織の俊敏制が大幅に向上される。
- わずか数分で世界にデプロイ:
awsを利用することによって数クリックで世界におリージョンにアプリケーションを世界に展開できる。


## 1/13
## AWS　well - architected フレームワーク
- AWSとそのパートナー会社がシステム設計と運用の経験から得た知見を六つの柱に基づいてまとめたベストプラクティス集。  
  1.オペレーショナルエクセレンスの柱(運用上の優秀性):システムの実行、モニタリング、プロセスの手順の継続的な改善に着目。  
  主なトピック:変更の自動化、イベントへの対応
  要するに業務プロセスについての視点で改善していくこと

  2.セキュリティの柱:情報システムの保護に焦点を当てている。  
  主なトピック:データの機密性と完全性、ユーザー許可の管理。

  3.信頼性の柱:期待通りに機能するか障害時すぐに回復できるか。  
  主なトピック:分散システムの設計、復旧計画、変化する要件への処理方法。

  4.パフォーマンス効率の柱:itやコンピューティングリソース(cpuなど)の構造化および合理化された割り当てに重点をおく。  
  主なトピック:最適化されたリソースタイプやサイズの選択、ビジネスニーズの増大に応じて効率を維持すること。

  5.コストの最適化の柱:不要なコストの回避に重点をおいている。  
  主なトピック:スケーリングなど。

  6.持続可能性の柱:環境への影響を最小限に抑えることに重点をおく。  
  主なトピック:持続可能性の責任共有モデル。

- AWS　well - architected　tool
  利用者の設計をベストプラクティクスに沿って評価して改善するための支援を行う。
- Design for Failure    
  故障に備えた設計という考え方をアーキテクチャに加えるということ。
        
   1.一つのデータセンターのみで運用しないようにする。        
   2.一つのハードウェアのみで構成しないようにする。

 ## 1/23
 ##　エッジロケーション  
 用途  コンテンツの低レイテンシー配信  
       可用性と拡張性の高いDNSの実現  

 - コンテンツの低レイテンシー配信
   CDNサービスであるamazon cloud Frontがエッジロケーションで利用できる。
   オンプレミスで配置せずAWSではエッジロケーションを利用することで簡単に低レイテンシー配信ができる。
   
 - 可用性と拡張性の高いDNSの実現
  　Amazon Route３がエッジロケーションで利用できる。またこれは高い可用性で利用できる。

 - Amazon　cloud FrontやRoute 53の保護
   DDosに対する保護サービス(AWS shield standerd)の対象で追加料金なしで保護される。
   Amazon　cloud Front はWAFによる保護もある。


 ## その他の多種多様な選択
 - Amazon Local zone
   Regionがない地域で低レイテンシーが必要な場合利用できる都市もある。
   Regionとは別の場所に作られたデータベースクラスタ。

 - AWS wavelength
   5Gネットワークが必要なアプリケーションなどに使われる。
   入ると5Gネットワークから外に出られない。

 - Amazon outposts
   AWSで使用している同じラックやハードウェアをユーザーが指定したデータセンターに配置できる。

##　AWSサービスへの認証とアクセスの管理
## AWSアカウントの保護
- AWSのアクセスには大きく分けて２つある  
1.ルートユーザー  
2.IAM(ユーザー、ロール、ポリシー)
  
|アクセス方法|説明|保護方法|使用頻度|
|---|---|---|---|
|ルートユーザー|Eメールとパスワードによってサインインするユーザーで,操作に制限がない|強固なパスワードで保護してMFAを設定する|低|
|IAM(indentity認証 and Access Manegiment認可|IAMユーザー、IAMユーザーグループ、IAMロール、IAMポリシーの要素で構成されている|-|高|  

## IAMユーザーとIAMポリシー
- IAMユーザーにAIMポリシーをアタッチしてできることを制限する。
  ※必要最低限の権限
- ３種類のIAMポリシー  
  
|名前|説明|アタッチ範囲|  
|---|---|---|
|AWS管理ポリシー|AWSが管理作成しているポリシーで初めからある|複数のIAMユーザー|  
|カスタマー管理ポリシー|ユーザーが作成管理しているポリシー|複数のIAMユーザー|  
|インフラポリシー|ユーザーが管理作成しているポリシー|直接指定できる|  

- IAMユーザーの認証情報  
  ユーザー名＋パスワード　→ マネジメントコンソール　→　API  
                又は　or　かつ    
  アクセスキー　＋　シークレットアクセスキー　→ CLI 又は　SDK → API
